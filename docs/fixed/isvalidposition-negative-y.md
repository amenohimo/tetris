# Bug: 壁蹴り後にピースが動けなくなり突然ゲームオーバー

## サマリ

回転（壁蹴り）でピースがボード上端より上にはみ出たとき、回転は成功したのに落下が拒否され、ピースがその場で固まって隠し行にブロックが溜まり、突然ゲームオーバーになるバグ。

**原因**: 回転用と移動・落下用で `isValidPosition` の「ボード上にはみ出し」に対する挙動が異なっていた。

**修正**: Board.ts の `isValidPosition` を Piece.ts に合わせ、`by < 0`（ボード上端より上）のセルをスキップするようにした。

---

## 症状

- プレイヤーから見ると「突然ゲームオーバーになった」
- 盤面にはまだ空きがあるように見える
- I ピースの回転直後に発生しやすい

## 再現条件

1. ボードが中盤〜終盤まで積まれている
2. I ピースが出現
3. 壁際で回転 → 壁蹴りによりピースが `y=-2` 付近に浮く
4. そのまま落下できず、ロックタイマー満了 → 隠し行に固定
5. 次のピースが spawn できず game over

---

## 関連するゲームメカニクス

### spawn（ピース出現）
ピースが新しくボード上部に現れること。I ピースは `y=-1`（隠し行よりさらに上）から出現し、スーッと降りてくる演出になっている。これは、ボードの一番上の行に突然現れるより自然に見えるため。

### 隠し行（hidden rows）
ボード上部の 2 行（`y=0〜1`）はプレイヤーに見えない予備エリア。ピースはここを通って画面内に降りてくる。隠し行にブロックが詰まると、次の spawn ができず game over になる。

### ロックタイマー（lock delay）
ピースが接地してから固まるまでの 500ms の猶予。接地後にプレイヤーが微調整できるようにする慈悲の時間。移動・回転のたびにリセットされる（最大15回）。

**今回のバグでは**: ピースが隠し行のさらに上（`y=-2`）にあり、`isValidPosition` が `y<0` を拒否したため、落下しようとしても「枠外」と判定されて動けず、あたかも接地しているかのようにロックタイマーが発動した。

### isValidPosition（位置の有効判定）
「このピース、ここに置いていいですか？」をチェックする関数。以下の2つを検査する：

1. ボードの枠からはみ出ていないか
2. すでに置いてあるブロックと重なっていないか

**この関数が 2 つ存在していた**。回転用（Piece.ts）と移動・落下用（Board.ts）。両者の基準が異なっていたことが根本原因。

### 壁蹴り（wall kick）
SRS（Super Rotation System）の仕様。壁際で回転するとき、そのままでは回れないため、自動でピースを少しズラす。横にズラすだけでなく、**上に浮かせる**キックも存在する。この浮きが今回のバグの引き金になった。

## トレースから見るバグの流れ

### トレースとは

game over 時に直近の操作履歴を出力するデバッグ機能。Console と画面に表示され、COPY ボタンでクリップボードにコピーできる。

### トレースの読み方

```
#フレーム番号 アクション種別  詳細

#7240 SPAWN      I at (3,-1)     ← Iピースが座標(3,-1)に出現
#7240 LOCK       Z at (7,16)     ← Zピースが(7,16)で固定、0ライン消去
#7289 ROT-L      I kick(0,-1) r0→r3  ← Iピースが左回転、キックoffset(0,-1)、回転0→3
#7323 SPAWN      O FAIL at (4,0) ← Oピースの出現に失敗
#7323 LOCK       I at (3,-2) FAIL ← Iピース固定でFAIL、次のspawn失敗
#7323 GAMEOVER   lock: spawnPiece after lock failed
```

| アクション | 意味 |
|---|---|
| `SPAWN` | ピースが出現（成功/FAIL） |
| `LOCK` | ピースが固定（ライン消去数、FAIL=spawn失敗） |
| `←` `→` `↓` | 左右移動・ソフトドロップ |
| `HARD` | ハードドロップ（落下行数） |
| `ROT+R` `ROT-L` | 右回転・左回転（キックoffset、回転前→後） |
| `HOLD` | ホールド（交換したピース種） |
| `GAMEOVER` | ゲームオーバー理由 |

### 実際のトレース（発生時）

```
#7240 SPAWN      I at (3,-1)       ← Iピース出現（y=-1: 隠し行の上から）
#7240 LOCK       Z at (7,16) lines=0

#7289 ROT-L      I kick(0,-1) r0→r3  ← 左回転。壁蹴りで y が -1 → -2 に

#7323 SPAWN      O FAIL at (4,0)   ← 34フレーム後（約0.5秒）、Oが出現できない
#7323 LOCK       I at (3,-2) lines=0 FAIL  ← I が (3,-2) で固定されたがspawn失敗
#7323 GAMEOVER   lock: spawnPiece after lock failed
```

**流れ**:

1. フレーム 7240: I ピースが `(3,-1)` に出現
2. フレーム 7289: プレイヤーが左回転。壁蹴り `kick(0,-1)` が成功し、ピースの位置が `y=-2` に
3. フレーム 7289〜7323 の間（約 0.5 秒）: 重力がピースを落とそうとするが、落下用 `isValidPosition` が `y<0` を拒否するため動けない。ロックタイマーが発動
4. フレーム 7323: ロックタイマー満了 → ピースが `(3,-2)` で固定。隠し行のさらに上なので画面には見えない位置にブロックが書かれる。直後の spawn で O ピースを産もうとするが、隠し行が塞がって失敗 → game over

### トレースなしでは何もわからなかった

トレース導入前は「突然 game over になった」という報告しかなく、再現も困難だった。

トレース導入により、以下の情報が明らかになった：

1. **回転イベントが見える**: `ROT-L kick(0,-1) r0→r3` から、壁蹴りで y 座標が -1 → -2 に移動したことがわかる
2. **Spawn 失敗の位置**: `SPAWN O FAIL at (4,0)` から、O ピースがボード上のどこで衝突したか特定できる
3. **時系列の把握**: フレーム番号から、回転から game over までの時間（約 0.5 秒 = ロックタイマー相当）が読み取れる
4. **GAMEOVER の理由**: `lock: spawnPiece after lock failed` から、ロック後の spawn で死んだことがわかる

## 根本原因

ボードの位置判定関数 `isValidPosition` が2つ存在し、挙動が異なっていた：

| | 回転用（Piece.ts） | 移動・落下用（Board.ts） |
|---|---|---|
| `y < 0`（ボード上端より上） | **許可**（スキップ） | **拒否**（return false） |
| 使われる場所 | `tryRotate()`（壁蹴り判定） | `tryMove()`, `tryMoveDown()`, `spawnPiece()` |

**Piece.ts の判定**（許可）:
```typescript
// ボード上にはみ出たセルは判定スキップ
if (boardY >= 0 && board[boardY][boardX] !== null) {
    return false;
}
```

**Board.ts の判定**（修正前、拒否）:
```typescript
// by < 0 は即座に拒否
if (bx < 0 || bx >= COLS || by < 0 || by >= ROWS) return false;
if (board[by][bx] != null) return false;
```

この不一致により：
- **回転時**: Piece.ts の判定が「上にはみ出てもOK」としたため、壁蹴りでピースが `y=-2` に移動できた
- **落下時**: Board.ts の判定が「上にはみ出し禁止」としたため、ピースがそこから動けなかった
- 動けない → ロックタイマー発動 → 隠し行に固定 → spawn 失敗 → game over

---

## 修正内容

Board.ts の `isValidPosition` を Piece.ts に合わせた。

`src/game/Board.ts`:

```diff
- if (bx < 0 || bx >= COLS || by < 0 || by >= ROWS) return false;
- if (board[by][bx] != null) return false;
+ if (bx < 0 || bx >= COLS || by >= ROWS) return false;
+ if (by >= 0 && board[by][bx] != null) return false;
```

- `by < 0` の即時拒否を削除 → 上にはみ出たセルを許可
- `board[by][bx]` のアクセスの前に `by >= 0` ガードを追加 → 負のインデックスでの配列アクセスを防止

### なぜ安全か

1. 「上移動」の操作は存在しない（`tryMove` は `dy >= 0` でのみ呼ばれる）
2. `lockPiece` は既に `by >= 0` ガード済み（隠し行より上にはブロックを書かない）
3. SRS のキックオフセットは最大でも ±2 セル以内に収まるため、ピースが無限に上に行くことはない
4. spawn 時の判定には影響しない（I ピースは `y=-1` だが全セルが `y=0` 以降にあるため）
